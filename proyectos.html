Modelo de Regresión Logística para predecir un 2.º ataque cardiaco

La idea de este proyecto es generar un modelo de Machine Learning para predecir si una persona va a tener un segundo ataque cardiaco basándonos en estudios realizados en el dataset.

Usamos Regresión Logística debido a que la variable objetiva "tener un 2.º ataque cardiaco" es binaria Si/No

Regresión Logística en RapidMiner:

Si seguimos el modelo de izquierda a derecha empezamos por la carga de datos basándonos en el dataset provisto con el operado Retrieve renombrado a Dataset estudios C.

Seguido de eso usamos el operado "Split Data" para generar las particiones de entrenamiento 70% y Test 30% fundamentales para que el modelo represente la realidad adecuadamente.

Luego usamos el operador de "Logistic Regresion" a modo de caja negra y le ingresamos la partición de entrenamiento y junto al operador "Apply Model" con la partición de test probamos el modelo.

Finalmente, multiplicamos la salida con el operador "Multiply" y mandamos el resultado del modelo al operador "Performance" para ver su precisión. El otro hilo lo mandamos hacia la salida result junto con los resultados de performance y ejecutamos el modelo.

Resultados predicción del modelo en RapidMiner:

Podemos ver en la 2 columna se encuentra el valor de predicción correcto y en la 3 se encuentra la predicción del modelo. A su vez vemos que la predicción depende de los coeficientes de Si (4 columna) y coeficiente de NO (5 columna) todo calculado a partir de los pesos de cada característica de las columnas restantes.

Resultados de precisión del modelo en RapidMiner:

Podemos ver que la precisión de este modelo es de 92.68% lo cual es bastante bueno, pero no es Perfecto. A su vez vemos que en dentro de los errores hay 2 falsos SI y 1 falso NO según la matriz de confusión.

Proyecto de Naibe Bayes para predecir condiciones para jugar al Golf

La idea de este proyecto es generar un modelo de machine learning para predecir basándonos en las condiciones climáticas del campo de golf si podemos practicar o no el deporte.

Si bien podríamos haber usado nuevamente Regresión Logística debemos usar Naibe Bayes debido al volumen limitado de datos de 14 tuplas.

Modelo de Naibe Bayes en RapidMiner:

El modelo consiste del dataset de entrenamiento "Golf", el dataset de test "Golf-TestSet", ambos son tratados por el operador "Numerical to polinomial" que convierte las características numéricas en polinomiales.

Luego el dataset de entrenamiento se usa para entrenar la caja negra de "Naive Bayes" y se aplica el modelo en "Apply Model" con el dataset de test.

Por último se mide la eficiencia con el operador Performance y presentamos los resultados hacia afuera.

Resultados predicción del modelo en RapidMiner:

Podemos ver en la 2 columna se encuentra el valor de predicción correcto y en la 3 se encuentra la predicción del modelo. A su vez vemos que la predicción depende de los coeficientes de Si (4 columna) y coeficiente de NO (5 columna) todo calculado a partir de los pesos de cada característica de las columnas restantes.

Performance del modelo con Naibe Bayes en RapidMiner:

En comparación al proyecto de regresión lineal la precisión es menor, pero considerando la poca cantidad de datos es un buen resultado, ya que el modelo no se desajustó.

Performance del modelo con Naibe Bayes en RapidMiner sin convertir las variables numéricas a polinomiales:

Al dejar las características numéricas sin convertirlas a polinomiales podemos ver que la precisión decae drásticamente empeorando el modelo y reflejando la importancia de las tareas de preparación de datos.

Proyecto de KNN para predecir deporte primario de un deportista

La idea de este proyecto es aplicar el algoritmo KNN para generar un modelo de machine learning.

La elección de knn se da para este modelo por la poca cantidad de datos, 447 tuplas y porque la variable objetiva es multivariable pudiendo ser uno de 4 deportes posibles y necesitando calcular un coeficiente para cada uno de estos deportes.

Modelo de KNN en RapidMiner:

El modelo consiste del dataset sport que pasa por el operador "Filter Examples" para eliminar outliers incoherentes.

Luego se setea el rol de "label" a la característica deporte primario para que sea la variable objetiva del dataset.

Finalmente, se manda al operador "Cross Validation" para correr el modelo varias veces mejorando un poco la eficiencia de este modelo.

Cross Validation del modelo en RapidMiner:

En el "Cross Validation" corremos varias veces el algoritmo de KNN como caja negra seguido de "Apply model" y "Performance" para medir la precisión.

Por último sacamos todo hacia afuera para ver los resultados.

Resultados predicción del modelo en RapidMiner:

Podemos ver en la 2 columna se encuentra el valor de predicción correcto y en la 3 se encuentra la predicción del modelo.

A su vez vemos que la predicción depende de los coeficientes de que están en la 4 a 7 columnas cada una calcula el coeficiente para cada deporte posible.

A partir de la 8 columna están las características de los deportistas.

Resultados de precisión del modelo en RapidMiner:

Por lo que vemos la performance es terrible, pero hay que considerar que son muchos deportes posibles, la cantidad de datos es poca y aun con todas las mejoras las predicciones no son buenas.

Para mejorarlo habría que poner muchos más datos para ajustar el modelo, a la par hay que mejorar la preparación de los datos seleccionando las características más significativas y quitando más outliers.

Proyecto Seleccion de Características

En este proyecto queremos comparar diferentes métodos de selección de características para ver los pros y cons de cada uno usando RapidMinner y un dataset con datos de un sonar para piedras y minerales.

Cross Validation adentro de cada Caja Negra de cada Selección:

Modelo de Naive Bayes aplicado al dataset con cada tipo de Selección de características adentro del Cross Validation:

Performance de Forward Selección:

Es el más balanceado de los 3 no es el más lento ni más rápido, pero tampoco es el más preciso o impreciso

Performance de Backwards Selección:

Muy rápido de ejecutar, pero fue el más impreciso

Performance de Selección con Algoritmo Genético:

Fue el más lento de ejecutar de los 3, pero es el método más preciso

Basándonos en este proyecto podemos concluir que dependiendo de la cantidad de datos y la precisión buscada debemos cambiar el algoritmo de selección de características para mantener la eficiencia.
